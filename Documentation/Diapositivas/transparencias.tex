\subsection{Programación paralela en dispositivos gráficos}
\begin{frame}
\frametitle{Programación paralela en dispositivos gráficos}
\begin{itemize}
 \item La programación paralela en dispositivos gráficos intenta explotar las ventajas de las
GPUs frente a las CPUs de propósito general. Para ello utiliza el paralelismo que ofrecen sus
múltiples núcleos, que permiten el lanzamiento de un altísimo número de hilos simultáneos. 
 \item Las aplicaciones se diseñan para que utilicen numerosos hilos que realizan tareas
independientes. 
 \item OpenCL es un framework que permite a los programadores implementar algoritmos para su
ejecución en GPU. 
\item OpenCL consta de un lenguaje de programación, un compilador y un conjunto de
herramientas. 
\end{itemize}
\end{frame}

\subsection{Algoritmo Shell Sort}
\begin{frame}
\frametitle{Shell Sort}
\begin{itemize}
 \item Algoritmo de ordenamiento se basa en una modificación del algoritmo de inserción en base a
las siguientes observaciones:  
\begin{enumerate}
  \item El Ordenamiento de inserción es eficiente si el algoritmo está ``casi ordenado''. 
  \item El ordenamiento es ineficiente, en general, porque mueve los valores una sola posición. 
 \end{enumerate}
 \item El algoritmo Shell sort mejora el ordenamiento por inserción comparando elementos separados
por un espacio de varias posiciones. Esto permite que un elemento haga comparaciones entre
posiciones más separadas y lo produce que que los elementos se muevan más rápido hacia su posición
esperada
 \item El algoritmo realiza múltiples pasos sobre los datos. Cada paso se hacen con tamaños de
espacio o incrementos más pequeños. 
\end{itemize}

\end{frame}

\subsection{Paso 1, un hilo por subconjunto, Implementación A}
\begin{frame}
 \frametitle{Paso 1, un hilo por subconjunto, Implementación A}
 \begin{itemize}
  \item Shell Sort se divide en pasos. Cada paso ordena un subconjunto de elementos independientes.
  \item En cada paso se lanza un hilo por subconjunto. 
  \item Ejemplo para 100 elementos, incrementos de Sedgewick (1, 5, 19, 41): 
  \begin{enumerate}
   \item 41 hilos, 2-3 elementos por hilo. 
   \item 19 hilos, 5-6 elementos por hilo. 
   \item 5 hilos, 20 elementos por hilo. 
   \item 1 hilos, 100 elementos por hilo. 
  \end{enumerate}

 \end{itemize}
\end{frame}
\subsection{Paso 2, varios hilos por subconjunto, Implementación B}
\begin{frame}
 \frametitle{Paso 2, varios hilos por subconjunto, Implementación B}
 \begin{itemize}
  \item El número de hilos de la implementación A se reduce en cada fase y el número de elementos
que ordena cada hilo es mayor. 
  \item Para mantener constante el número de hilos lanzados, esta implementación lanza más de
  un hilo por subconjunto que lo ordenan parcialmente. 
  \item Los datos no se ordenan por completo porque no existe comunicación entre los hilos
  que ordenan el mismo subconjunto de elementos.
  \item Ejemplo para 100 elementos, incrementos de Sedgewick (1, 5, 19, 41), cada hilo ordena al
menos 4 elementos: 
  \begin{enumerate}
   \item 41x1 (41) hilos, 2-3 elementos por hilo. 
   \item 19x2 (38) hilos, 4 elementos por hilo (último hilo de cada subconjunto ordena 1-2
elementos). 
   \item 5x5  (25) hilos, 4 elementos por hilo. 
   \item 1x25 (25) hilos, 4 elementos por hilo. 
  \end{enumerate}
 \end{itemize}
\end{frame}
\subsection{Paso 3, Corrección de la implementación B, Implementación C}
\begin{frame}
 \frametitle{Paso 3, Corrección de la imp. B, Implementación C}
 \begin{itemize}
  \item En la implementación B no existen comunicación entre los hilos.
  \item Se añaden comparaciones entre los extremos de los datos que gestiona cada hilo. Si no se
encuentran bien colocados entre ellos se intercambian y vuelven a ordenarse cada subconjunto. 
  \item Como los elementos están ``casi ordenados'', no se espera un número de repeticiones muy
alto. 
 \end{itemize}

\end{frame}

\subsection{Paso 4, Copia en local de los datos, Implementación D}
\begin{frame}
 \frametitle{Paso 4, Copia en local de los datos, Implementación D}
 \begin{itemize}
  \item Se intenta mejorar el rendimiento del algoritmo realizando copia privada de los datos que
  maneja cada hilo.
  \item Cuando cada hilo ordena sus elementos, copian el primer y último de sus datos en memoria
  global. 
  \item Se comparan los datos de memoria global, y si es necesario se intercambian los valores,
  copiando estos en memoria privada. 
 \end{itemize}

\end{frame}

\section{Conclusión}
\begin{frame}
 \tableofcontents[currentsection]
\end{frame}

\begin{frame}
 \frametitle{Conclusión}
 \begin{itemize}
  \item ShellSort es un algoritmo que para ordenar los datos, divide en cada fase el conjunto
  de datos en varios subconjuntos que ordena independientemente. Esta característica favorece su
  implementación en dispositivos gráficos. 
  \item Sin embargo, las últimas fases, al reducir el número de subconjuntos, hacen que el
  rendimiento del algoritmo en paralelo no sea tan bueno. 
  \item Las diferencias entre las fases también afectan negativamente al rendimiento paralelo de 
  las implementaciones paralelas. Mientras que en las primeras fases, la cantidad de datos que se
  ordena cada subconjunto es muy pequeña y se encuentran muy separadas, según van avanzando las
  fases los datos son más y se encuentran mas cercanos. 
 \end{itemize}
\end{frame}
\begin{frame}
 \frametitle{Implementación final}
 \begin{itemize}
  \item La implementación final no consigue mejorar de forma significativa el tiempo de ejecución
  en CPU. 
  \item Los datos obtenidos no son definitivos debido a la diferencia de prestaciones entre
las CPU y los dispositivos gráficos de los equipos utilizados. 
 \end{itemize}
\end{frame}
